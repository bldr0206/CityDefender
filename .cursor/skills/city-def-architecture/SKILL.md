# Архитектура City Def

## Назначение

Один источник правды для агента о том, **как устроен этот репозиторий**: сцены, папки скриптов, роли `Game` / `Actions`, контент уровней. Детальные правила глобальных статиков — в skill `unity-global-statics` (личный); здесь — только привязка к City Def.

## Границы ответственности агента (экономия контекста)

**Делать:** C#-скрипты (`.cs`), шейдеры (например `.shader`, `.hlsl`, при необходимости — код под Shader Graph), краткая **документация** по задаче: комментарии в коде, при необходимости небольшой `.md` в репозитории (чеклист настройки, что повесить на объект, какие поля заполнить).

**Не создавать и не править** без явной просьбы пользователя: префабы (`.prefab`), материалы (`.mat`), сцены (`.unity`), ассеты на диске для `ScriptableObject` (`.asset`), настройки рендера/постобработки как YAML-ассеты, Animator, Timeline как файлы, произвольные бинарные/проприетарные ассеты Unity. Ручная расстановка объектов в сцене, привязка ссылок в инспекторе, создание префабов — **зона автора**: это дорого по токенам и удобнее в редакторе.

**Как передавать результат:** публичные `[SerializeField]` / поля под назначение в инспекторе, имена слоёв и тегов в комментарии, пошаговый список «создай пустой объект → повесь компонент → перетащи …». Если без ассета никак — описать структуру словами; не генерировать YAML префабов и сцен «вслепую».

Исключение: пользователь **прямо** попросил создать или изменить конкретный ассет — тогда по запросу.

## Сцены и поток

- **Meta** — мета-экран (кнопка Play и т.д.), переход в бой через `SceneManager.LoadScene` (см. `MetaScreenLogic`).
- **Battle** — боевая сцена.

Новые точки входа в игру или смена порядка сцен — править соответствующую логику мета/загрузки и держать имена сцен согласованными с `EditorBuildSettings` (в Unity).

## Раскладка скриптов (`Assets/_Game/Scripts`)

| Папка | Назначение |
|--------|------------|
| `Meta/` | UI и логика мета-сцены (`MetaScreenLogic` и дальнейшие экраны меты). |
| `Static/` | Статические хабы: `Game` (состояние), `Actions` (события и вызовы). Один класс — один файл. |
| `ScriptableObjects/` | Данные контента, например `LevelsList` с `LevelData` (имя, префаб уровня). |

Префабы и арт — под `Assets/_Game/` (Prefabs, Content, Scenes и т.д.); новые системы класть по смыслу рядом с фичей, не плодить дублирующие «глобальные» статики вне `Game` / `Actions`.

## `Game` и `Actions`

- **`Game`** — планируемое единое место для устойчивого состояния забега/сессии (уровень, валюта, флаги и т.д.). Сейчас класс может быть почти пустым — новое состояние добавлять сюда осознанно, с узким API (свойства/методы), а не размазывать по MonoBehaviour.
- **`Actions`** — единственная публичная точка для игровых событий: объявление события, метод вызова, подписка снаружи только на этот API. Не заводить параллельные `static Action` для того же смысла в других типах.

Подробные правила и чеклист — skill **unity-global-statics**.

## Контент уровней

- **`LevelsList`** (`ScriptableObject`) — список `LevelData` (строковое имя, ссылка на префаб уровня). Ассет лежит в контенте проекта (например `Assets/_Game/Content/`). Новый уровень: данные в SO + согласованность с загрузчиком, когда он появится.

## Конвенции Unity в этом проекте

- Один `MonoBehaviour` на файл; для нового UI-текста — Text Mesh Pro (skill **unity-conventions**).
- В C# по договорённости проекта — **регионы** для крупных логических блоков в больших файлах; без XML-комментариев к публичному API, если не оговорено иначе.

## Чеклист при изменениях

- [ ] Новое «глобальное» состояние — в **`Game`** (или явное решение не трогать глобальный слой).
- [ ] Новый сквозной сигнал — через **`Actions`**.
- [ ] Мета / сцены — **`MetaScreenLogic`** и связанные скрипты, имена сцен не расходятся с билдом.
- [ ] Данные уровней и префабы — **`LevelsList`** и контент-папки, без дублирования списков уровней в трёх местах.
- [ ] Новый функционал без лишних ассетов от агента: только код/шейдер/инструкция; сцены и префабы настраивает автор, если не просили иначе.
